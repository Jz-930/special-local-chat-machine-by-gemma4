# 4-29 仪表盘改造方案

## 背景

当前 Open WebUI 的“上下文统计 / context load”仪表盘实际使用字符数计算：

- 记忆区：`($manualMemoryText || '').length`
- 历史消息：`content.length + 20`
- 总量：`contextCharCount = vaultCharCount + historyCharCount`
- UI：`(contextCharCount / 10000).toFixed(1)` 显示为 `W`

这会导致仪表盘看起来像 token 负载，但实际是 UTF-16 code units 字符数。对中文长文本、混排文本、Markdown、代码块、标点和模型 tokenizer 差异来说，这个口径明显不准。

目标不是增加大量“估算”说明，而是让仪表盘的数字本身更接近真实 prompt token 负载。

## 改造目标

1. 仪表盘主数字改为 token load，而不是字符数。
2. Navbar、MemoryDrawer、Chat payload 使用同一套上下文统计逻辑。
3. 根据当前选中模型使用不同 token profile，提高中文长文本场景准确度。
4. 利用 provider 返回的真实 `usage.input_tokens` / `prompt_tokens` / `prompt_eval_count` 做后验校准。
5. 不引入重型 tokenizer 依赖，不阻塞发送流程，不显著增加前端性能成本。

## 总体方案

采用“真实 token anchor + delta 估算 + 启发式 token 计算 + 模型 profile + 自动校准”的方案。

第一阶段不直接接入真实 tokenizer，而是实现一个足够接近的前端 token estimator：

- 对 CJK、英文、数字、空白、标点、Markdown/code block 分别加权。
- 每条 message 加固定 role/chat 模板 overhead。
- 根据模型 id 选择不同 profile。
- 如果后续真实请求返回 usage，则把这次真实 prompt token 作为 anchor 快照。
- 后续上下文没有发生结构性变化时，优先用“上次真实 prompt token + 新增/删除内容 delta”计算当前负载。
- 如果 anchor 不可信，再退回到“全量估算 * 模型校准倍率”。

这样可以把当前“字符数伪装成 token”的误差显著压低，而且实现成本和运行成本都可控。

## 三层计算策略

仪表盘每次刷新时按以下优先级计算：

```ts
if (hasValidAnchorOnCurrentChain) {
	return estimateByActualAnchorPlusDelta();
}

if (hasUsableCalibrationForModel) {
	return estimateFullContext() * calibrationFactor;
}

return estimateFullContext();
```

三层含义：

1. **Anchor + Delta**：最优先。使用上一次 provider 返回的真实 prompt token 作为锚点，只估算此后新增、删除、修改的部分。
2. **全量估算 * 校准倍率**：当找不到可靠 anchor，但该模型已有历史校准倍率时使用。
3. **纯全量估算**：开始阶段或没有任何可靠历史数据时使用。

这可以避免长上下文场景里每次都从零估算。长 Vault、长历史这类最容易累积误差的部分由真实 usage 锚住，前端只估算变化量。

## 模型 Profile

## 文本分段规则

`estimateTextTokens()` 不使用 `.length` 直接算总数，而是遍历 Unicode code point：

- CJK：`\p{Script=Han}`、Hiragana、Katakana、Hangul
- Latin：连续 `[A-Za-z]`
- 数字：连续 `[0-9]`
- 空格 / Tab：按压缩后的空白估算
- 换行：单独计入
- 标点：单独计入
- 代码块：检测三反引号区域，使用 `codeCharsPerToken`

伪代码：

```ts
function estimateTextTokens(text, profile) {
	let tokens = 0;
	for (const segment of segmentText(text)) {
		if (segment.type === 'cjk') tokens += segment.count * profile.cjkTokenRatio;
		if (segment.type === 'latin') tokens += segment.count / profile.latinCharsPerToken;
		if (segment.type === 'number') tokens += segment.count / profile.numberCharsPerToken;
		if (segment.type === 'punctuation') tokens += segment.count * profile.punctuationTokenRatio;
		if (segment.type === 'newline') tokens += segment.count * profile.newlineTokenRatio;
		if (segment.type === 'space') tokens += segment.count / profile.whitespaceCharsPerToken;
		if (segment.type === 'code') tokens += segment.count / profile.codeCharsPerToken;
	}
	return Math.ceil(tokens);
}
```

## Navbar 改造

当前变量：

- `contextCharCount`
- `vaultCharCount`
- `historyCharCount`
- `maxContextLoad`

建议改为：

- `contextTokenCount`
- `vaultTokenCount`
- `historyTokenCount`
- `maxContextTokens`

`maxContextLoad` 的 localStorage key 建议迁移：

- 旧：`maxContextLoad`
- 新：`maxContextTokens`

迁移逻辑：

1. 如果存在 `maxContextTokens`，优先使用。
2. 如果不存在但存在旧 `maxContextLoad`，把旧值作为 token 阈值沿用一次。
3. 写回新 key。

仪表盘显示：

- 主显示：`{(contextTokenCount / 10000).toFixed(1)}W`
- 阈值：`/ {maxContextTokens / 10000}W`
- 进度条：`contextTokenCount / maxContextTokens`

`parseSmartLimit()` 改成 token 阈值解析：

- 所有输入最终都归一化成 tokens。
- UI、localStorage、进度条比较值都只保存 token 数，不保存字符数。

支持输入：

- `250000` => `250000 tokens`
- `250000 tokens` / `250000 token` / `250000 tok` => `250000 tokens`
- `128k` => `131072 tokens`
- `256k` => `262144 tokens`
- `25w` / `25万` => `250000 tokens`
- `1m` => `1048576 tokens`
- `25万字` / `25w 字` / `250000 chars` => 按当前模型 profile 换算成 tokens 后保存

也就是说，`k`、`m` 默认按模型上下文窗口常用的 1024 进位解释，`w`、`万` 按十进制万解释；只有用户显式写了 `字`、`字符`、`chars`、`characters` 时，才先解释为字符数量，再换算为 tokens。

## 单位归一化规则

新增函数：

```ts
export function parseContextBudgetToTokens(input: string, modelId?: string): number | null;
```

解析逻辑：

1. 先识别数量级：
   - `k` = `1024`
   - `w` / `万` = `10000`
   - `m` = `1048576`
2. 再识别单位类型：
   - token 单位：`token`、`tokens`、`tok`、`t`
   - 字符单位：`字`、`字符`、`char`、`chars`、`character`、`characters`
   - 无明确单位：默认按 tokens 处理
3. 如果是 token 单位或无明确单位，直接返回 token 数。
4. 如果是字符单位，使用当前模型 profile 把字符预算换算成 token 预算。

字符预算换算不需要真实文本，但要按模型 profile 的典型中文写作场景估算：

```ts
function convertCharacterBudgetToTokens(charBudget, profile) {
	return Math.ceil(charBudget * profile.cjkTokenRatio);
}
```

原因：这个项目主要上下文压力来自中文长文本和 Vault 设定，字符预算按 CJK profile 换算比继续按 1:1 更可靠。

示例：

- 当前模型为 Qwen，`25万字` => `250000 * 1.05 = 262500 tokens`
- 当前模型为 Llama，`25万字` => `250000 * 1.35 = 337500 tokens`
- 当前模型未知，`25万字` => `250000 * 1.25 = 312500 tokens`

保存时写入：

```ts
localStorage.setItem('maxContextTokens', String(parsedTokens));
```

不再写入字符阈值。后续无论用户曾经输入什么单位，仪表盘都只拿 token 数参与计算和显示。

## MemoryDrawer 改造

当前 `totalLength` 是各 card title/content 的 `.length` 相加，这和实际注入 prompt 的文本不一致。

应改为：

1. 使用 `compileCardsToText(cards)` 得到真实注入 Vault 文本。
2. 对该文本调用 `estimateTextTokens()`。
3. 卡片列表里的单卡统计也用 token estimator，而不是 `(card.content || '').length`。

注意：单卡统计可以只算 title + content；底部总量必须算 `compileCardsToText(cards)`，因为标题标记和分隔符也会进 prompt。

## Chat.svelte 校准方案

Chat.svelte 构造 payload 的位置已经会把 Vault 拼入 system prompt：

```ts
const manualVault = get(manualMemoryText).trim();
if (manualVault !== '') {
	systemPromptContent = `...${manualVault}...` + systemPromptContent;
}
```

改造点：

1. 在 `messages` 最终构造完成后，调用 `estimateContextTokens()` 得到本次发送前的本地 token 估算。
2. 把估算值暂存到 response message 上，例如：

```ts
responseMessage.contextTokenEstimate = {
	model: model.id,
	inputTokens: estimate.total,
	hasFiles: files?.length > 0,
	hasTools: selectedToolIds.length > 0,
	hasWebSearch: selectedFilterIds.length > 0
};
```

3. 响应完成时，如果 `message.usage` 有真实输入 token，则更新模型校准倍率。

真实输入 token 取值优先级：

```ts
usage.input_tokens
usage.prompt_tokens
usage.prompt_eval_count
```

## Anchor 快照与 Delta 估算

每次响应完成后，如果 provider 返回真实输入 token，并且本次请求没有文件、RAG、web search、工具等前端不可见注入，就建立一个 anchor 快照。

anchor 表示：“生成这个 response message 时，真实 prompt token 是多少。”

注意：上一轮 `actualPromptTokens` 是生成上一条 assistant 回复时的输入 token，通常不包含这条 assistant 回复本身。因此下一轮计算当前上下文时，要额外把 anchor 之后新增进入上下文的消息算进去，尤其是上一条 assistant response。

### 快照结构

```ts
type ContextTokenAnchor = {
	modelId: string;
	chatId?: string;
	anchorMessageId: string;
	actualPromptTokens: number;
	rawEstimatedPromptTokens: number;
	messageChainHash: string;
	messageChainIds: string[];
	systemPromptHash: string;
	systemEstimatedTokens: number;
	vaultTextHash: string;
	vaultEstimatedTokens: number;
	settingsHash: string;
	payloadFlags: {
		hasFiles: boolean;
		hasTools: boolean;
		hasWebSearch: boolean;
		hasRag: boolean;
	};
	createdAt: number;
};
```

建议把 anchor 挂在 response message 上，同时在 localStorage 保存最近若干个 anchor：

```ts
localStorage.setItem('contextTokenAnchors', JSON.stringify({
	[chatId]: {
		[messageId]: anchor
	}
}));
```

不要只保存一个全局 anchor。倒回、分支、切聊天时都需要沿当前 message chain 找最近可用 anchor。

### Delta 计算

如果当前链路上能找到可用 anchor：

```ts
currentTokens =
	anchor.actualPromptTokens
	+ estimateMessagesAfter(anchor.anchorMessageId)
	+ vaultDelta
	+ systemDelta
	+ templateDelta;
```

其中：

- `estimateMessagesAfter(anchor.anchorMessageId)`：当前 message chain 中 anchor 之后新增进入 prompt 的消息。
- `vaultDelta = currentVaultEstimatedTokens - anchor.vaultEstimatedTokens`
- `systemDelta = currentSystemEstimatedTokens - anchor.systemEstimatedTokens`
- `templateDelta`：消息数量变化造成的 role/chat 模板 overhead。

如果 Vault 或 system prompt 改动不大，delta 会比整段重新估算准确；如果改动很大但 hash 可比，仍然可以用估算 delta 修正。

### Anchor 有效性校验

anchor 必须满足：

1. `modelId` 一致。
2. `anchorMessageId` 在当前 message chain 上。
3. `settingsHash` 一致，或变化项可以明确做 delta。
4. 本次 anchor 来源不是文件、工具、RAG、web search 请求。
5. 当前 payload 结构仍然由前端可见消息、system prompt、Vault 构成。

如果任一条件不满足，丢弃该 anchor，继续向祖先链找更早的 anchor。找不到就退回全量估算。

### 指纹 Hash

不要在每个 UI 操作里写复杂特判。统一通过 hash 判断上下文结构是否变化。

建议 hash：

```ts
messageChainHash = hash(
	messageChain.map(m => ({
		id: m.id,
		role: m.role,
		contentHash: hash(extractMessageText(m.content)),
		parentId: m.parentId
	}))
);

systemPromptHash = hash(systemPromptContent);
vaultTextHash = hash(compiledVaultText);
settingsHash = hash({
	stripThink: currentStripThinkValue,
	memoryValve: currentMemoryValveValue
});
```

`hash()` 可以使用项目已有依赖 `js-sha256`。

## 易错场景处理

这些场景不应靠分散的事件监听处理，而应由“找当前链路最近 anchor + hash 校验 + delta”统一处理。

| 场景 | 处理方式 |
|---|---|
| 开始的时候 | 没有真实 anchor，走全量估算。第一次响应返回 usage 后建立 anchor。 |
| 倒回的时候 | 当前链路变短或切到祖先消息。沿当前祖先链找最近 anchor；找不到就全量估算。 |
| 开辟不同分支 | 每条 response message 都可以有自己的 anchor。切分支时按当前 `currentId` 的祖先链查找。 |
| 开关记忆防爆阀 | 如果影响实际 payload，纳入 `settingsHash`；hash 变了则旧 anchor 失效，或按明确规则做 delta。 |
| 开关某个记忆卡片 | `vaultTextHash` 改变，用 `vaultDelta` 修正。 |
| 编辑记忆卡片 | 重新 `compileCardsToText(cards)` 后计算 `vaultDelta`。 |
| 导入/删除记忆卡片 | 同上，使用新的 Vault 文本与 anchor 中的旧 Vault token 对比。 |
| 切换模型 | `modelId` 不一致，anchor 不可用。不同模型 tokenizer 不能混用。 |
| 修改 system prompt | `systemPromptHash` 改变。能计算旧新 system token 差时走 `systemDelta`，否则 anchor 失效。 |
| 开关 strip thinking | 纳入 `settingsHash`。它会改变 assistant reasoning 是否计入上下文，变化后旧 anchor 失效。 |
| 文件 / RAG / web search / 工具调用 | 可以显示真实 usage，但不建立 anchor、不更新校准倍率，避免后端不可见注入污染估算。 |
| 后端 filter/outlet 改写消息 | 前端历史同步后 message hash 会变化，旧 anchor 自动失效或回退到更早 anchor。 |
| 临时聊天 | 前端直接发送完整 messages，anchor + delta 可靠度最高。 |
| 持久聊天 | 后端可能从 DB 组装历史，必须校验 `chatId + anchorMessageId + messageChainHash`；校验不上就全量估算。 |

## 自动校准

新增 localStorage key：

- `contextTokenCalibrationByModel`

自动校准和 anchor 是两套机制：

- anchor：优先用于当前聊天链路，基于真实 prompt token 做 delta。
- calibration：没有可用 anchor 时，用于修正全量估算。

结构：

```json
{
  "qwen3:235b": {
    "factor": 1.12,
    "samples": 4,
    "updatedAt": 1714350000000
  }
}
```

更新规则：

```ts
newFactor = actualInputTokens / estimatedInputTokens
factor = oldFactor * 0.7 + newFactor * 0.3
```

限制：

- `newFactor` 小于 `0.5` 或大于 `2.5` 时丢弃，避免异常污染。
- 有文件、工具、RAG、web search、后端额外注入内容时不校准。
- estimatedInputTokens 小于 `1000` 时不校准，小样本误差太大。
- 如果本次响应能建立 anchor，可以同时更新 calibration；但 calibration 只作为 anchor 不可用时的兜底。

Navbar 使用：

```ts
const calibrationFactor = getCalibrationFactor(selectedModelId);
const displayedTokens = Math.round(rawEstimatedTokens * calibrationFactor);
```

实际 Navbar 应优先读取 `estimateContextTokens()` 返回的 mode：

```ts
const estimate = estimateContextTokens(...);

if (estimate.mode === 'anchor_delta') {
	displayedTokens = estimate.total;
} else if (estimate.mode === 'calibrated_estimate') {
	displayedTokens = estimate.total;
} else {
	displayedTokens = estimate.total;
}
```

UI 不需要暴露复杂模式；模式主要用于调试和后续排查。

## 预期效果

改造后：

- 中文长文本不会再按“1 字 = 1 token”硬算。
- Llama/Mistral/Gemma 这类中文 token 更贵的模型会自动显示更高负载。
- Qwen/DeepSeek 这类中文 tokenizer 较友好的模型会显示较低负载。
- 有历史真实 usage 的聊天会优先使用 anchor + delta，长上下文误差不会随总长度线性累积。
- 倒回、分支、记忆卡片开关、记忆卡片编辑等场景通过 message chain 和 Vault hash 自动处理。
- 常用模型经过几次真实 usage 校准后，仪表盘会逐步贴近实际 prompt token。
- 用户看到的 `20W / 25W` 可以按 token 负载理解，而不是字符数。

## 不做事项

第一版不做以下内容：

- 不接入真实 tokenizer WASM。
- 不为每个 provider 单独请求 token count API。
- 不在每次输入变化时异步请求后端计算 token。
- 不把文件、RAG、工具返回内容强行计入前端仪表盘。

这些可以作为第二阶段优化，但第一阶段先保证仪表盘从“字符数”升级为“相对准确的 token 负载”。
