# 本地私有化 AI 大模型架构搭建日志 (Special Local Chat Machine)

## 项目背景
**目标**：探索与搭建本地私有化大语言模型 (LLM) 推理架构，主要应用于长文本小说生成与实验。并构建一套长期扩展的综合生态库（包含大语言模型前端、潜在后端脚本、微调项目等）。
**核心选型**：
- **资源底座**：RunPod 云端算力实例部署 (租用双 RTX 6000 级或其它具有 96GB 庞大显存池的资源)。
- **前端交互界面**：基于本地部署的 `Open-WebUI` 进行深度改造及对接使用。
- **大核心模型**：采用 `llmfan46/gemma-4-31B-it-uncensored-heretic`，主打深度的逻辑关联及无删减自由度，符合小说的完全发散和创意需求。
- **量化方案**：“GGUF 格式” 的 Q8 (`Q8_0`) 半无损量化版。平衡显存利用：将 31B 结构压缩至 ~32GB 的极致体积下，从而释放高达 50GB+ 以上的 GPU 显存专供 KV Cache，承接十万甚至几十万量级的上下文(Tokens)记忆流。

### 阶段二：服务器推算与基建框架切换决策
**时间**：2026-04-23
**状态**：✅ 已完成
**面临抉择（vLLM vs Ollama）：**
- 最初考量使用纯计算速度最顶级的 `vLLM` 框架容器。但在具体对接指定 HuggingFace 量化库时触发重大安全隐患：指定的目标库包含近 200GB 由各种不同量化级（Q3至F32）构成的大杂烩，常规 vLLM 云模板极易触发全库盲目下载，撑爆 100GB 磁盘并导致死锁瘫痪。
- 后调整为兼具优雅与定点拉取能力的 `Ollama` 容器作为后端。一方面，Ollama 现已内置基于外链的点对点下载指令，只拉取绝对核心的 `Q8_0` 的 32GB 单体文件；另一方面，Open-WebUI 的根血统即基于 Ollama，两者在交互扩展体验上拥有 100% 同构的顺滑亲和力。

**学习意义**：明白前后端隔离的本质——云主机就是一块插在远端的“网络大显卡”，它越纯粹，留给小说的长记忆越充沛；同时云端防线需要针对性开口（用 0.0.0.0 和端口暴露机制）以连接本地。

### 阶段四：实机 400 异常排查与非标模型强制组装
**时间**：2026-04-23
**状态**：✅ 组装成功
**踩坑记录**：
尝试使用傻瓜指令 `ollama run hf.co/xxxx` 自动从 HuggingFace 抓取时，遇到极其隐蔽的 `Error: 400 Bad Request` 中断BUG。
- **问题探明**：非官方定制的 GGUF 库往往存在清单结构 (Manifest) 格式不完全符合最新规范的情况。引发的后果是：32GB 的权重哪怕被全数下载完毕，最后一步协议封包时也会因为识别不到参数而立刻崩溃报废。
- **强制接管措施 (双指令绕行硬解)**：
  1. 通过提取底层网络工具直接劫持文件流，绕开平台验证和 Registry 解析，在根目录直接硬写入 32.6GB 的实体大发：
     `curl -L [...]/resolve/main/[...]-Q8_0.gguf -o /workspace/models/gemma.gguf`
  2. 写一份专用的指令文本 (Modelfile)： `FROM /workspace/models/gemma.gguf` 指引出它的本地坐标。
  3. 通过 `ollama create gemma-novel -f /.../Modelfile` 完成本地非基准组装，重塑出属于我们自己的可用模型。
**学习意义**：永远不要迷信傻瓜一键脚本。掌握纯手动的 "实体文件极客下载 -> 编写特征文件 -> 本地硬重塑" 三段式装配手法，才是未来驾驭任何开源魔改模型的底牌。

**学习意义**：本地环境绝不是一股脑儿地 `pip install` 和全局安装环境污染。明确业务核心（只写小说），就能放手剔除不需要的工业级冗余包，实现既轻量又安全的沙盒化本地开发流。

### 阶段七：底层框架休克性断代排查与热裁剪
**时间**：2026-04-23
**状态**：✅ 修复完成
**问题描述**：在本地使用前沿版本 `Python 3.14.2` 启动环境时，相继触发了两次致命性的依赖中断（服务端完全卡死）。
1. **Pydantic V1 底层不兼容**：由于用于 RAG 处理的本地向量库 `chromadb` 锁定依赖了陈旧的 `pydantic.v1`，其实例化操作在 Python 3.14 的反射机制下引起了全局 ConfigError 阻断。
2. **Audioop 模块彻底淘汰**：核心路由 `audio.py` 请求了涉及发音模块的依赖包 `pydub`，而由于 Python 官方在 3.14 中彻底移除了服务于此的 `audioop` 标准库，造成启动时 `ModuleNotFoundError`。
**物理级裁剪措施 (Hot-Fixing)**：
- **剥离 ChromaDB 调用栈**：定位到后端 `config.py` 和 `retrieval/vector/factory.py` 内部，将触发崩溃的 `import chromadb` 进行阻拦，并通过重写派生自 `VectorDBBase` 的虚假客户端模型（DummyClient），在物理层面截断这块闲置组件。
- **软熔断 Pydub 引用**：将 `routers/audio.py` 的关键依赖包袱通过 `try-except ImportError` 软性熔断设置为 `None`。完全摘除对于我们不需要的语音引擎检测代码。
- **修复脚本调度丢失机制**：排查指出，原 `dev_start.bat` 在调动 `venv\Scripts\activate.bat` 后未采用跨脚本挂起命令 `call`，导致其在激活环境指令下发后，后段包含 uvicorn 引导的执行序列被进程强制截胡销毁。随即通过补全 `call` 指令全盘闭环。

### 阶段九：前端推理死锁修复与附属冗余请求切除
**时间**：2026-04-23
**状态**：✅ UI 卡顿彻底解决
**问题描述**：在成功生成对话文字后，WebUI 前端的左侧侧边栏会陷入无休止的“转圈加载”（Deadlock）。
**原理解析**：
Open-WebUI 在新对话产生的首回合后，会在后台偷偷派生出两个子请求（Secondary Requests）：要求模型自动生成一个短句作为“标题”及“分类标签”。由于我们使用的是高度发散、未经指令服从微调的 `gemma-novel (uncensored)` 模型，模型彻底无视了简短归纳的指令，输出了长篇大论或混乱格式，导致前端的 JSON 解析器陷入无限期等待。
**干预措施**：
- 进入 `dev_start.bat`，在环境启动序列中注入高压管制项：`set ENABLE_TITLE_GENERATION=False` 与 `set ENABLE_TAGS_GENERATION=False`。
- 直接在引擎级别瘫痪这两种伴生请求。这不仅立刻治愈了前端侧边栏的死循环卡顿，还为本地书写节省了至少一半由于生成毫无意义标题所浪费的云端算力。

### 阶段十一：“钢印级”手工隔离记忆抽屉 (Memory Vault)
**时间**：2026-04-23
**状态**：✅ 完成部署
**功能痛点**：在长篇小说的跨章节漫长对决里，模型即便上下文再庞大，也会因为注意力分散或滑窗滚动机制（Sliding Window Pruning）而遗忘前文的核心世界观定律。
**技术突破与落地**：

- **防止对话串台（哈希防沉切）**：记忆库的存取与当前的动态 `$chatId` 严密结合，不同的历史对话（或不同部小说）在浏览器层面彼此独享保险箱，老死不相往来；并建立“未落定 ID 首推防丢机制”。
- **绝对穿透与监控器**：在源码 `sendMessageSocket` 即将发射前置入拦截器，把抽屉内部的信息附上“小说后台记忆法则：最高优先级”标签，以一号 System 权重粗暴塞入队列顶端。针对 256K 超大杯特性，在页脚实装 5 万字软监控哨塔，仅做变色提示从不强制截断。

### 阶段十二：绝密提取！零污染“幽灵总结器” (Ghost Summary)
**时间**：2026-04-23
**状态**：✅ 验证通过且工作完美
**功能痛点**：若用户直接在聊天内要求大模型概括目前已推移的进度大纲，此类问答行为将会产生劣质的“助理问答上下文（Assistant-QA Context）”，严重破坏模型沉浸式讲述小说的写作口吻。
**超限外延式架构**：
- 于页面顶部导航挂载魔法大纲组件 `<GhostSummaryModal />`。
- 它的工作原理犹如黑客截取通讯：前端纯函数 `createMessagesList` 提取目前最后 12 轮（24 发消息）情节，独立绕开主线的 JSON 处理体系，封装一个自杀性质只进不退的新生 API Payload：
  `"请用一段话精炼、客观地总结上述内容的剧情进展，不准评价角色，禁止废话！"`
- 此过程产生的总结在黑板弹窗内生成后即算销毁，不再被写回主线的记忆深库。用户可以快速复印到刚建成的记忆抽屉内，彻底免去了反复阅读推敲的脑力负担。

**状态**：✅ PWA 原生体验达成
**功能说明**：系统的物理架构基于 Tailscale 内网穿透技术延伸到了移动端，为了满足手机/平板窄屏下无盲区的单手写作体验，对三神器进行了断点式的 CSS（Tailwind `sm:` 体系）下沉降级：
- **灵动暗码 (Responsive Sequence IDs)**：打破了在 PC 宽屏下依赖外部边距悬浮的排版规律，当检测到纯手机视口时，行号绝对坐标（如 `#0001`）自动向内靠拢吸附于气泡右上方。同时鉴于触摸屏无 Hover 特性的物理法则，将其脱离触发器的基础透明度直接翻倍。
- **降维折叠仪表盘 (Flex-Column Stacking)**：大幅重写魔杖提取器内部水平锁死的标尺框栏位，在移动端自动换行断点并重构为下压弹夹式（Stacking）的两层设计，将精密输入框与快捷宏大面积隔开，彻底根除了拇指误触率与界面挤压越界的隐患。
- **全屏原生拟态**：得益于浏览器对 PWA 的“添加至主屏幕”支持以及右侧悬浮栏 (`MemoryDrawer`) 自带 320px 全高占位的抽屉级兼容，实现等价原生 App 无边框的高沉浸移动端网文创作舱。

### 阶段十五：架构进阶与网络共享盘 (Network Volume) 漂移
**时间**：2026-04-23
**状态**：✅ 架构升级成功
**架构痛点与动机**：
先前方案中，GGUF 巨型权重文件锁死在 RunPod 单一实例附带的临时硬盘 (Volume Disk) 中。一旦因配置变更或更换算力节点（例如从 RTX 6000 切换到 A100）导致原实例被销毁，32GB 的模型实体就会灰飞烟灭，需要重新忍受漫长的部署与下载周期。
**破局与重构措施 (The Floating Workspace)**：
全面转接使用该数据中心的 **Network Volume (网络共享存储)**。
1. **暗渡陈仓式的路径挂载**：建立 60GB 的 Network Volume。在启动 Pod 的高级模板中，放弃网络盘默认的 `/runpod-volume`，强制手写 **Volume mount path = `/workspace`**。以此达成“移花接木”，保证了前期积累的所有硬编码本地命令行脚本可做到“0 感知”、无损继承。
2. **锁死模型下沉路径**：植入核心变量 `OLLAMA_MODELS=/workspace/models`，逼迫 Ollama 将模型装配打散后产生的所有持久化结构强制写进属于我们的网络盘私域中。
3. **超集性能引擎参数**：直接加入 `OLLAMA_FLASH_ATTENTION=1` 彻底接管长篇小说级（十万字）续写过程中的 KV Cache 显存海啸泄漏；并加入 `OLLAMA_KEEP_ALIVE=-1` 建立张量记忆体永驻机制。极大地拉升了发呆、断点重连情况下的反应极速。
**实战排坑再记录 (The Missing `curl`)**：
在全新建立的实例使用旧把戏“硬物理接管下载 (curl 绕行)”时，遭遇极简化 `ollama` 原生无头镜像不支持网络爬行工具的致命剥离 (`bash: curl: command not found`)。
顺畅衔接强拆三段式流程，顺利将模型主体化为永不磨灭的数据流固定于云端。

### 阶段十六：记忆防爆阀 (Reasoning Stripper) 与卡片化设定库重构
**时间**：2026-04-24
**状态**：✅ 核心上下文控制体系升级
**架构痛点与动机**：
在使用 Gemma-4-31B-it (Thinking 模型) 进行长篇小说创作时，由于其强大的深度逻辑推理会在每轮对话中产生长篇的 `<think>` 标签块。随着对话轮数增加，几十轮的 `<think>` 被累积缝合进历史上下文，会导致极其宝贵的 256k Token 额度迅速枯竭，引发灾难性的记忆截断。同时，旧版的全局“设定面板”由于采用单一长文本框，当世界观设定超过 20,000 字时，编辑与管理变得寸步难行。

**破局与重构措施**：
1. **构建“记忆防爆阀”拦截器 (Reasoning Stripper)**：
   - 彻底摒弃前端直接屏蔽的妥协方案，直接在 Open-WebUI 后端的 `ollama.py` 和 `openai.py` 路由中植入底层正则拦截器。
   - 当大模型准备读取历史消息前，拦截器以毫秒级速度“物理抹除”所有历史助手消息中的 `<think>...</think>` 内容，确保给 LLM 输入纯净的小说正文上下文。
2. **防爆阀的 UI 化与动态控制 (The Time-Machine Toggle)**：
   - 考虑到某些高难度情节依然需要大模型的深层逻辑，我们将拦截器升级为可通过前端 UI 实时热更的动态开关（顶部导航栏粉色剪刀图标）。
   - 由于前端 IndexedDB 会永久保留所有 `<think>`，因此在小说遇到创作死结时，可以随时将开关打向 **OFF**。下一秒，大模型就会“瞬间回想起”前几十轮中被阉割的内心独白与推理过程，完美实现“局部全景智力爆发”。
   - 将该开关状态绑定 Svelte Store 和 LocalStorage，做到不同聊天房间（Chat ID）状态物理隔离，互不干扰。
3. **彻底重构：卡片化设定与记忆面板 (Card-Based Memory Vault)**：
   - 修复了此前“新建空草稿会覆盖历史数十万字设定”的竞态条件 Bug。
   - 将原先笨重的单页长文本框 UI 推翻，重构为具备“独立标题、字数统计、折叠概览、一键删改”的 **卡片化 (Card-Based)** 管理矩阵。
   - **自动无损迁移**：系统会自动抓取旧对话中的长文本，封装进《导入的旧设定》卡片中，实现零痛点升代。
   - **底层拼接优化**：重构了发包逻辑，将零散卡片按 `### 标题\n内容\n\n---\n\n` 的严密 Markdown 语法进行拼装并注入。这种结构化数据不仅解放了人类的双手，更使得 LLM 能够以更高权重去解析和遵循复杂的角色与世界观规则。

### 阶段十七：上下文全景仪表盘与极限前端优化
**时间**：2026-04-24
**状态**：✅ 性能瓶颈突破与 UI 升维
**功能说明**：随着控制工具的增多，闭眼“盲飞”容易导致 256k 上下文在不知不觉中被写爆。同时，前端浏览器在渲染长篇网文历史记录时出现了严重的滚动卡顿。
1. **实时上下文载荷仪表盘 (Context Load Dashboard)**：
   - 在顶部导航栏植入了一个具备**双行显示**与**动态能量条**的高级进度模块 (`Navbar.svelte`)。
   - 极其精准的底层对齐：加入了 System Prompt 系统开销 (~50字) 与 ChatML 对话格式开销 (~20字/轮) 的精准计算。
   - **防爆阀联动**：当粉色防爆阀开启时，仪表盘会以毫秒级速度利用正则 `/<think>[\s\S]*?(?:<\/think>|$)/gi` 将历史对话中庞大的思维链剔除，进度条会肉眼可见地瞬间回落，提供绝对的安全感。
   - 设置了 20万字（黄标警告）与 25万字（红光呼吸灯警告）的两级防爆阈值。
2. **记忆设定卡片“热插拔” (Memory Vault Hot-Swapping)**：
   - 拒绝一刀切的删除，在每张记忆卡片上新增了 iOS 风格的滑动开关。
   - 关闭（卸载）卡片时，触发视觉幽灵化（降透明度、置灰、增加删除线）。同时底层发包系统和仪表盘会立刻扣除该卡片的字数，实现特定章节设定的灵活隐身。
3. **前端 DOM 极限渲染优化 (Visual Thinking Collapse)**：
   - **问题定位**：前端滚轮卡顿并非因为没有懒加载，而是因为视口内的几条历史消息一旦包含上万字的 `<think>`，就会导致浏览器的 Markdown 引擎 (Marked.js) 被海量的 AST 节点强行撑爆内存。
   - **外科手术式截断**：拦截了 `ContentRenderer.svelte` 的渲染入口，只要某条历史消息已生成完毕，且开启了防爆阀，系统就会在文字进入解析器前，将其物理蒸发并替换为一行极简提示：`> 🧠 深度思考过程已自动折叠`。
   - **零代价的流畅**：成功卸载了 99% 的无效 DOM 节点，让长篇对话的上下滑动如丝般顺滑。由于拦截只在前端显示层，后端的 IndexedDB 原文和上下文抓取不受任何影响。
