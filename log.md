# 本地私有化 AI 大模型架构搭建日志 (Special Local Chat Machine)

## 项目背景
**目标**：探索与搭建本地私有化大语言模型 (LLM) 推理架构，主要应用于长文本小说生成与实验。并构建一套长期扩展的综合生态库（包含大语言模型前端、潜在后端脚本、微调项目等）。
**核心选型**：
- **资源底座**：RunPod 云端算力实例部署 (租用双 RTX 6000 级或其它具有 96GB 庞大显存池的资源)。
- **前端交互界面**：基于本地部署的 `Open-WebUI` 进行深度改造及对接使用。
- **大核心模型**：采用 `llmfan46/gemma-4-31B-it-uncensored-heretic`，主打深度的逻辑关联及无删减自由度，符合小说的完全发散和创意需求。
- **量化方案**：“GGUF 格式” 的 Q8 (`Q8_0`) 半无损量化版。平衡显存利用：将 31B 结构压缩至 ~32GB 的极致体积下，从而释放高达 50GB+ 以上的 GPU 显存专供 KV Cache，承接十万甚至几十万量级的上下文(Tokens)记忆流。

**时间**：2026-04-23
**状态**：✅ 已完成
**面临抉择（vLLM vs Ollama）：**
- 最初考量使用计算最顶级的 `vLLM`。但指定的 HuggingFace 库集成了近 200GB 由各种不同量化级构成的大杂烩，常规 vLLM 极易触发全库盲目下载，引发 100GB 磁盘爆表导致瘫痪死锁。
- 后调整为兼具精确定向下载能力的 `Ollama` 容器作为底座。且 Open-WebUI 本就脱胎于 Ollama 体系，同构亲和力 100%。

### 阶段三：云端容器镜像与网络环境精准配置
**时间**：2026-04-23
**状态**：✅ 部署成功
**关键动作**：
1. **容器模板**：弃用冗杂的 UI 版模板，采用极致纯粹的 **`Ollama NVIDIA CUDA (ollama/ollama:latest)`**，确保云端只做 API 算力引擎，内存 100% 服务核心。
2. **网络通行与持久化护城河 (四大避坑环境变量)**：
   - `暴露端口 11434`：提供外部通讯锚点。
   - `OLLAMA_HOST=0.0.0.0`：打破本地回环网络限制，允许用户开发机的请求打入。
   - `OLLAMA_MODELS=/workspace/models`：将超大体积权重强制转移到 120GB 主容量盘，避免撑爆 15GB 宿主机盘。
   - `OLLAMA_KEEP_ALIVE=-1`：强制禁止模型休眠释放机制，写小说停顿构思时保证永远 0 秒唤醒延迟。

### 阶段四：实机 400 异常排查与非标模型强制组装
**时间**：2026-04-23
**状态**：✅ 组装成功
**踩坑记录**：
尝试使用傻瓜指令 `ollama run hf.co/xxxx` 自动从 HuggingFace 抓取时，遇到极其隐蔽的 `Error: 400 Bad Request` 中断BUG。
- **问题探明**：非官方定制的 GGUF 库往往存在清单结构 (Manifest) 格式不规范（未按照 Ollama 的底层 Registry 体系配置）。引发的后果是：32GB 的权重哪怕被全数下载完毕，最后一步协议封包时也会立刻崩溃报废。
- **强制接管措施 (双指令绕行硬解)**：
  1. 通过安装 `curl` 直接劫持下载底层流，绕开平台验证和 Registry 解析，在根目录直接硬写入 32.6GB 的实体文件：
     `curl -L [...]/resolve/main/[...]-Q8_0.gguf -o /workspace/models/gemma.gguf`
  2. 写一份专用的指令文本 (Modelfile) `FROM /workspace/models/gemma.gguf` 指引出它的文件坐标。
  3. 通过 `ollama create gemma-novel -f /.../Modelfile` 完成本地非基准组装，赋予专属代号。
**学习意义**：永远不要迷信傻瓜一键抓取脚本。在开源世界里，掌握纯手动的 "实体文件溯源下载 -> 编写特征文件 -> 本地硬重塑" 三段式装配手法，才是驾驭任何魔改模型的万金油绝招。
