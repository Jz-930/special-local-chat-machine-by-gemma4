# 4-30 档案功能管理方案

## 目标

围绕“碎片库”和“记忆抽屉”建立一套精确、可批量、可热插拔的个人档案管理系统。

核心目标有三类：

1. 碎片库支持一层分组，方便大量碎片的收纳、筛选和查看。
2. 碎片库、记忆抽屉、系统记忆支持批量导入、批量迁移和批量删除。
3. 所有个人使用信息可以导出为根目录下的一个档案包文件夹，也可以从该文件夹一键导入，实现不依赖账号密码的“热插拔”。

## 当前实现判断

根据当前代码结构，相关数据大致分成四类：

1. 碎片库
   - 当前存在一条 Notes 记录中，标题为 `MemoryVault_Global_Fragments`。
   - 内容是卡片数组，序列化后放在 `note.data.content.md`。
   - 管理页面在 `src/routes/(app)/fragments/+page.svelte`。
   - 记忆抽屉中也会读取这条全局碎片库记录。

2. 记忆抽屉
   - 每个对话对应一条 Notes 记录，标题格式为 `MemoryVault_<chatId>`。
   - 内容同样是卡片数组，放在 `note.data.content.md`。
   - 还有 localStorage 兜底，键名包括 `manualMemoryVAULT_cards_byChat` 和旧格式 `manualMemoryVAULT_byChat`。
   - 主要逻辑在 `src/lib/components/chat/MemoryDrawer.svelte`。

3. 系统记忆
   - 存在后端 `memory` 表中，字段包括 `id/user_id/content/created_at/updated_at`。
   - API 在 `/api/v1/memories`。
   - 系统记忆会同步写入向量库，导入后需要重新生成 embedding 或调用重建接口。

4. 对话与归档对话
   - 对话在 `chat` 表中，归档状态由 `archived` 字段控制。
   - 详细消息同时存在 `chat.chat` JSON 和 `chat_message` 表中。
   - 已有接口支持导入对话、导出统计、查询归档对话、归档/取消归档、删除对话等。

结论：方案可行。碎片库和记忆抽屉可以先沿用 Notes 作为存储层，较小改造即可支持分组与批量操作；热插拔档案包需要新增后端导入导出服务，把 chats、notes、memories、folders/tags/settings 等数据统一打包，并在导入时处理 ID 映射和向量库重建。

## 需求完善

### 1. 碎片库一层分组

只允许一层分组，不做多级目录。

建议交互：

- 左侧或顶部显示分组列表：全部、未分组、用户自定义分组。
- 每个碎片只能属于一个分组。
- 支持创建、重命名、删除分组。
- 支持分组颜色，用颜色辅助快速识别大量碎片。
- 支持分组手动排序，碎片也支持组内手动排序。
- 删除分组时不删除碎片，而是把碎片移入“未分组”。
- 支持批量移动碎片到某个分组。
- 支持按分组筛选、搜索标题/内容、按更新时间或手动排序。

建议数据结构：

```json
{
  "version": 2,
  "groups": [
    {
      "id": "group_worldbuilding",
      "name": "世界观",
      "color": "#3b82f6",
      "order": 10,
      "created_at": 1714435200,
      "updated_at": 1714435200
    }
  ],
  "fragments": [
    {
      "id": "fragment_id",
      "group_id": "group_worldbuilding",
      "title": "魔法体系",
      "content": "...",
      "enabled": true,
      "order": 20,
      "created_at": 1714435200,
      "updated_at": 1714435200
    }
  ]
}
```

兼容旧数据：

- 如果 `MemoryVault_Global_Fragments` 里还是旧数组格式，读取时自动包装成 `{ version: 2, groups: [], fragments: oldArray }`。
- 保存时统一写入新格式。
- 旧字段尽量保留，不强制破坏已有卡片。

### 2. 碎片库和记忆抽屉批量操作

#### 碎片库批量能力

列表页进入“选择模式”后支持：

- 批量删除碎片。
- 批量移动到分组。
- 批量导入到当前对话的记忆抽屉。
- 批量导出为 JSON 文件或加入档案包。
- 批量去重，默认按 `title + content` 或 `content` 归一化后判断。

#### 记忆抽屉批量能力

抽屉内支持：

- 多选卡片。
- 批量保存到碎片库。
- 批量从碎片库导入到当前抽屉。
- 批量删除当前抽屉卡片。
- 批量启用/停用卡片。
- 批量移动排序。

#### 碎片库与记忆抽屉互导规则

建议使用“复制导入”，而不是“移动导入”作为默认：

- 从碎片库导入记忆抽屉：生成新的卡片 ID，保留 `source_fragment_id`。
- 从记忆抽屉保存到碎片库：生成新的碎片 ID，保留 `source_chat_id` 和 `source_card_id`。
- 如果内容完全相同，默认覆盖已有记录或刷新来源信息，但必须先做一致性核查。
- 如果只是标题相同、内容不完全相同，不能自动覆盖，必须进入冲突预览。
- 一致性核查使用归一化内容 hash：去掉首尾空白、统一换行、可选忽略连续空格；同时显示原始内容差异入口。

示例字段：

```json
{
  "id": "card_id",
  "title": "主角背景",
  "content": "...",
  "enabled": true,
  "source_fragment_id": "fragment_id",
  "source_chat_id": "chat_id",
  "source_card_id": "old_card_id"
}
```

### 3. 系统记忆批量管理

当前 `/memories` 已支持单条新增、单条删除、清空全部、重建向量库。为了成批管理，需要新增：

- `POST /api/v1/memories/batch/add`
- `DELETE /api/v1/memories/batch`
- `POST /api/v1/memories/batch/import`
- `GET /api/v1/memories/export`

建议导入策略：

- 默认按内容去重。
- 支持 `skip_duplicates`、`replace_duplicates`、`append_all`。
- 导入后异步重建或增量写入向量库。
- 批量导入要避免一次性并发过多 embedding 请求，建议分批处理，例如每批 16 到 32 条。

## 热插拔档案包设计

### 基本概念

“档案包”是用户选择的磁盘目录下的一个文件夹，不依赖账号密码。这里的“根目录”不是服务器内部固定目录，而是导出/导入时通过目录选择器选中的本机目录、移动硬盘目录或 U 盘目录。

建议位置：

```text
archives/
  2026-04-30-my-profile/
    manifest.json
    chats/
    memory-vaults/
    fragments/
    memories/
    notes/
    folders/
    settings/
    files/
    logs/
```

一键导出：前端弹出系统目录选择器，让用户选择本机或 U 盘目录；后端生成档案包文件树，前端把文件写入用户选择的目录。

一键导入：前端弹出系统目录选择器，让用户选择包含 `manifest.json` 的档案包文件夹；前端读取文件并交给后端校验 manifest、预览差异，然后按策略导入当前实例。

一键清空：后端清空当前用户或当前工作区的使用资料，用于切换档案包前彻底卸载当前内容。清空必须是独立危险操作，不能和导入自动绑定。

热插拔 MVP 必须同时具备三件事：

- 从文件夹导入。
- 导出到文件夹。
- 清空所有使用资料。

### manifest.json

```json
{
  "schema_version": 1,
  "app": "open-webui-memory-vault",
  "exported_at": "2026-04-30T00:00:00-04:00",
  "export_mode": "current-user",
  "source": {
    "user_id": "source-user-id",
    "workspace_name": "local"
  },
  "counts": {
    "chats": 120,
    "archived_chats": 80,
    "memory_vaults": 90,
    "fragments": 300,
    "fragment_groups": 12,
    "memories": 40,
    "notes": 20
  },
  "checksum": {
    "algorithm": "sha256",
    "files": {
      "chats/chats.jsonl": "..."
    }
  }
}
```

### 推荐文件结构

```text
archives/<档案包名>/
  manifest.json
  chats/
    chats.jsonl
    archived-chats.jsonl
    tags.json
    folders.json
  memory-vaults/
    vaults.jsonl
    localstorage-backup.json
  fragments/
    fragments.json
  memories/
    memories.jsonl
  notes/
    notes.jsonl
  settings/
    user-settings.json
  files/
    attachments/
  logs/
    export-report.json
    import-report.json
    clear-report.json
```

### 对话导出内容

每条对话建议保留：

- 原始 `chat.id`
- `title`
- `chat` JSON
- `archived`
- `pinned`
- `folder_id`
- `meta`
- `created_at`
- `updated_at`
- tags
- 关联文件引用

导入时建议生成新 ID，并建立 `old_chat_id -> new_chat_id` 映射。这样可以避免覆盖当前实例已有对话。

### 记忆抽屉导出内容

导出所有标题符合 `MemoryVault_<chatId>` 的 Notes。

导入时需要根据对话 ID 映射处理：

- 如果对应对话也在本档案包里导入，则改写为 `MemoryVault_<new_chat_id>`。
- 如果只导入记忆抽屉，不导入对话，则不再创建未绑定抽屉，而是把这些卡片导入碎片库的特殊分组“孤立抽屉”。
- “孤立抽屉”分组是系统保留分组，可以重命名显示名，但内部 key 固定，便于后续继续识别。
- 孤立抽屉导入碎片库时，碎片需要保留 `source_vault_chat_id`、`source_vault_title`、`source_card_id`，方便以后追溯原对话来源。

### 普通 Notes 导出内容

普通 Notes 是热插拔 MVP 的必选范围。

导出规则：

- 导出用户正常创建的 Notes。
- 排除 `MemoryVault_<chatId>`，因为它们由“记忆抽屉”专用流程处理。
- 排除 `MemoryVault_Global_Fragments`，因为它由“碎片库”专用流程处理。
- 保留 note 的 `id/title/data/meta/is_pinned/created_at/updated_at/access_grants`。

导入规则：

- 默认生成新 note id，避免覆盖当前已有笔记。
- 如果标题和正文内容完全一致，允许执行完全一致覆盖或刷新元数据。
- 如果标题相同但内容不同，进入冲突预览，允许保留现有、覆盖、另存为副本。
- 如果用户模式停用，则导入时统一归属到当前运行用户或当前工作区内部 user_id。

### 碎片库导出内容

导出 `MemoryVault_Global_Fragments`，统一写为新结构：

```json
{
  "version": 2,
  "groups": [],
  "fragments": []
}
```

导入策略：

- `merge`：合并到现有碎片库，默认策略。
- `replace`：先备份现有碎片库，再替换。
- `append_as_group`：把整个档案包的碎片导入为一个新分组。
- `exact_overwrite`：当 hash 核查确认标题和内容完全一致时，允许覆盖或刷新已有记录的元数据。

冲突判断：

- 完全一致：`normalized_title_hash + normalized_content_hash` 一致，允许覆盖。
- 内容一致但标题不同：列入“同内容不同标题”，默认保留现有标题，允许用户批量改用导入标题。
- 标题一致但内容不同：列入“疑似冲突”，必须人工选择保留现有、覆盖、另存为副本。
- 内容近似但不完全一致：第一版不自动判断语义相似，只做文本 hash 和可选 diff。

### 系统记忆导出内容

导出 `memory` 表内容，不导出向量库索引作为主路径。

原因：

- 向量库和 embedding 模型强相关，跨机器或跨版本直接恢复索引容易不一致。
- 导入后重建 embedding 更稳定。

导入流程：

1. 写入 `memory` 表。
2. 按批生成 embedding。
3. upsert 到 `user-memory-<user_id>` 向量集合。
4. 生成导入报告。

### 文件与附件

第一版不纳入上传文件和知识库。

原因：

- 当前使用方式没有使用上传文件和知识库。
- 这些是 Open WebUI 原生功能，不属于当前记忆精确管理系统的核心路径。
- 如果界面里存在相关入口，第一版只需要隐藏或弱化入口，不需要做迁移。

后续如果要补齐完整迁移，再把附件和知识库作为第四阶段扩展：

- 导出文件元数据和实际文件内容到 `files/attachments/`。
- 复制文件到当前实例的存储目录。
- 生成新的 file id。
- 改写 chat JSON 中的文件引用。

## 安全和可靠性要求

热插拔功能涉及本机文件系统和后端导入解析，必须加限制：

- 主流程必须使用浏览器/桌面客户端目录选择器，由用户显式选择导出目录或导入档案包目录，支持 U 盘、移动硬盘和任意有权限的本机目录。
- 服务器暂存区只能作为高级备用能力，限制在配置允许的根目录中读写，例如 `DATA_DIR/archive-packs`。
- 后端必须做路径归一化，禁止 `../` 路径穿越。
- 导入前必须先 inspect，显示数量、版本、风险、冲突数量。
- replace 模式必须先自动生成备份包。
- 清空所有使用资料前必须先 inspect，并默认自动导出一个备份档案包。
- 清空、replace、删除档案包都必须二次确认。
- 二次确认不能只是普通弹窗；需要用户输入指定确认文本，例如档案包名或“清空所有使用资料”。
- 导入全程写入 `import-report.json`。
- 导出全程写入 `export-report.json`。
- 清空全程写入 `clear-report.json`。
- 对每个 JSON/JSONL 文件做 schema 校验。
- 大文件导入导出使用流式处理，避免一次性加载全部数据。
- 导入失败时尽量保证局部事务，不留下半写入的核心数据。

## 清单报告界面

导入、导出、清空都要有清单报告界面。

导出报告：

- 本次导出档案包路径。
- 导出范围勾选项。
- 普通对话数量、归档对话数量、普通 Notes 数量、记忆抽屉数量、碎片数量、碎片分组数量、系统记忆数量。
- 被跳过项目及原因。
- manifest 校验结果。
- 导出耗时和错误列表。

导入预览报告：

- 档案包基本信息：名称、创建时间、来源、schema 版本。
- 将导入的项目数量。
- 与当前数据的冲突数量。
- 完全一致可覆盖数量。
- 标题相同但内容不同数量。
- 孤立抽屉数量，以及将导入到“孤立抽屉”分组的碎片数量。
- 普通 Notes 冲突数量。
- 是否需要重建系统记忆向量。
- 本次导入策略：合并、替换、导入为新分组、完全一致覆盖。

导入完成报告：

- 成功导入数量。
- 覆盖数量。
- 跳过数量。
- 失败数量和失败原因。
- old_id 到 new_id 的映射文件位置。
- 系统记忆向量重建状态。

清空预览报告：

- 将清空的数据范围。
- 将删除的普通对话、归档对话、普通 Notes、记忆抽屉、碎片库、系统记忆数量。
- 清空前自动备份档案包名称。
- 明确提示不会清空的系统级内容，例如用户账号、应用配置、模型配置。

清空完成报告：

- 实际清空数量。
- 自动备份路径。
- 失败项目和原因。
- 后续可执行动作：导入档案包、返回空工作区、查看备份。

## 前端入口设计

建议新增一个“档案管理”页面，放在侧边栏或设置页中。

页面分为五个区域：

1. 档案包列表
   - 显示根目录下已有档案包。
   - 支持查看 manifest、导入、删除。

2. 一键导出
   - 可勾选数据范围。
   - 默认勾选：普通对话、归档对话、普通 Notes、记忆抽屉、碎片库、系统记忆。
   - 输出为一个目录，不强制 zip。
   - 导出结束后进入导出报告页。

3. 一键导入
   - 选择档案包。
   - 先预览，再导入。
   - 导入策略：合并、替换、导入为新分组/新命名空间、完全一致覆盖。
   - 合并和替换都要在预览页明确展示影响范围。
   - 替换必须二次确认，并要求输入档案包名。
   - 导入结束后进入导入报告页。

4. 清空所有使用资料
   - 独立危险操作入口，不能放在普通批量工具里。
   - 先生成清空预览报告。
   - 默认自动创建清空前备份。
   - 需要二次确认，并要求输入“清空所有使用资料”。
   - 清空结束后进入清空报告页。

5. 批量工具
   - 整理碎片库中的“孤立抽屉”分组。
   - 对碎片库去重。
   - 重建系统记忆向量。
   - 检查 chat 与 MemoryVault 绑定关系。

## 实施阶段

### 第一阶段：碎片库分组和批量操作

目标：先把日常管理效率提升起来，不改动太多后端。

任务：

- 把碎片库数据结构升级到 version 2。

## 关键设计取舍

1. 碎片库继续用 Notes 存储还是新建表？
   - 短期建议继续用 Notes，改动小，能快速上线。
   - 长期如果碎片规模很大，建议新建 `fragment_group` 和 `fragment` 表，避免大 JSON 频繁整体读写。

2. 热插拔导入是否保留原 ID？
   - 默认不保留，导入时生成新 ID，减少冲突。
   - manifest 中保留 old_id，导入报告保存 old/new 映射。

3. 导入是替换还是合并？
   - 默认进入预览页，由用户选择合并或替换。
   - 合并是推荐默认项。
   - 替换必须自动备份，并做二次确认。
   - 碎片库可提供“导入为新分组”，这是最符合团队协作的方式。

4. 是否导出向量库？
   - 默认不导出向量索引，只导出原始文本。
   - 导入后重建向量，更稳定。

5. 用户模式停用后如何归属数据？
   - 导入导出以“当前本地工作区”或“当前运行用户”为归属，不依赖账号密码。
   - 如果系统仍有内部 user_id，则导入时统一映射到当前 user_id。

## 已确认设计决策

1. 第一版不包含上传文件和知识库。相关入口如存在，先隐藏或弱化；普通 Notes 必须包含在热插拔范围内。

2. 热插拔必须包含三项核心操作：从文件夹导入、导出到文件夹、清空所有使用资料。

3. 合并和覆盖都可以提供，但必须先预览，再二次确认；覆盖和清空属于危险操作，必须有明确防呆。

4. 碎片库分组需要颜色和手动排序。

5. 完全相同内容应该覆盖，但必须做一致性核查；非完全一致不能自动覆盖。

6. 孤立抽屉不做独立未绑定列表，统一导入碎片库的“孤立抽屉”分类。

7. 导入、导出、清空都必须设计清单报告界面。

## 仍需实现时确认的细节

1. 清空所有使用资料是否清空基础 settings。建议默认不清空 settings，只清空对话、归档对话、普通 Notes、记忆抽屉、碎片库、系统记忆、folders/tags。

2. “完全一致”的 hash 是否只比较标题和内容，还是同时比较 group/source/enabled/order。建议覆盖判断只比较标题和内容，其他字段作为元数据刷新。

3. “孤立抽屉”分组是否允许用户删除。建议允许删除分组，但删除前必须提示会删除其中所有孤立抽屉碎片，或提供移动到其他分组选项。

## 建议优先级

推荐先做第一阶段和第三阶段 MVP：

1. 碎片库分组。
2. 碎片库/记忆抽屉多选和批量互导。
3. 完全一致覆盖核查。
4. 档案包导出核心数据，包括普通 Notes。
5. 档案包 inspect + 合并/替换导入。
6. 清空所有使用资料。
7. 导入/导出/清空清单报告。
8. 系统记忆导入后重建向量。

这样最先解决碎片爆炸和团队“文件夹热插拔”的核心痛点，同时避免一开始就被完整附件迁移、多用户权限、知识库文件等复杂问题拖慢。
