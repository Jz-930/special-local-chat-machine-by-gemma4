## 原问题

改造前，Navbar 仪表盘看起来像 token 上下文负载，但实际是字符数统计：

- Vault / 记忆区：`($manualMemoryText || '').length`
- 历史消息：`content.length + 20`
- 总量：`contextCharCount = vaultCharCount + historyCharCount`
- UI 显示：`(contextCharCount / 10000).toFixed(1)`，单位显示为 `W`

这个口径的问题是：

- JavaScript `.length` 统计的是 UTF-16 code units，不是模型 tokenizer token。
- 中文、英文、数字、标点、Markdown、代码块的 token 比例不同。
- Qwen / DeepSeek / Llama / GPT / Claude 等模型 tokenizer 对中文的切分差异很大。
- 用户看到 `20W / 25W` 时容易理解成 token 负载，但旧实现更接近“20 万字符 / 25 万字符阈值”。

本次改造目标不是增加 UI 注释，而是让数字本身更接近真实 prompt token。

## 当前统计原理

本次实现采用三层策略：

```ts
if (hasValidAnchorOnCurrentChain) {
	return estimateByActualAnchorPlusDelta();
}

if (hasUsableCalibrationForModel) {
	return estimateFullContext() * calibrationFactor;
}

return estimateFullContext();
```

### 1. Anchor + Delta

优先使用 provider 返回的真实 prompt token 作为锚点。

当某次响应完成后，如果后端返回了真实输入 token：

- `usage.input_tokens`
- `usage.prompt_tokens`
- `usage.prompt_eval_count`
- `usage.usage.input_tokens`
- `usage.usage.prompt_tokens`

前端会把这次真实 token 记录为 `ContextTokenAnchor`。

之后仪表盘刷新时，如果当前聊天链路上能找到有效 anchor，就使用：

```ts
currentTotal =
	anchor.actualPromptTokens
	+ introducedMessagesTokens
	+ vaultDelta
	+ systemDelta
```

其中：

- `anchor.actualPromptTokens`：上一次真实 prompt token。
- `introducedMessagesTokens`：anchor response 自身以及 anchor 之后新增进入上下文的消息 token。
- `vaultDelta`：当前 Vault token 与 anchor 时 Vault token 的差值。
- `systemDelta`：当前 system prompt token 与 anchor 时 system prompt token 的差值。

注意：provider 返回的上一轮 prompt token 通常不包含“上一轮 assistant 回复本身”。所以下一轮计算当前上下文时，需要把 anchor response 也算进 delta。

### 2. Calibrated Estimate

如果当前链路没有可用 anchor，但该模型已经有历史 usage 校准数据，则使用：

```ts
displayedTokens = rawEstimatedTokens * calibrationFactor
```

校准倍率存储在 localStorage：

```text
contextTokenCalibrationByModel
```

更新规则：

```ts
newFactor = actualInputTokens / estimatedInputTokens
factor = oldFactor * 0.7 + newFactor * 0.3
```

为了避免异常污染：

- 本次请求有文件、RAG、web search、工具调用时不更新 calibration。
- `estimatedInputTokens < 1000` 时不更新，小样本误差太大。
- `newFactor < 0.5` 或 `newFactor > 2.5` 时丢弃。

### 3. Full Estimate

没有 anchor、也没有 calibration 时，走全量 token 估算。

估算逻辑不再使用 `.length`，而是按 Unicode code point 遍历文本，并按类型加权：

- CJK：中文、日文假名、韩文等，按模型 profile 的 `cjkTokenRatio` 计算。
- Latin：连续英文字符按 `latinCharsPerToken` 压缩。
- Number：连续数字按 `numberCharsPerToken` 压缩。
- Whitespace：空格、tab 按 `whitespaceCharsPerToken` 压缩。
- Newline：换行单独计入。
- Punctuation：标点单独计入。
- Markdown syntax：额外增加轻量 overhead。
- Code block：三反引号代码块按 `codeCharsPerToken` 单独估算。
- Message overhead：每条消息增加 role/chat 模板 overhead。
- System overhead：system prompt 增加系统消息 overhead。
- Image part：按固定 image token cost 计入。

## 单位归一化原则

本次改造后，仪表盘内部只保存和比较 tokens。

localStorage 新 key：

```text
maxContextTokens
```

旧 key：

```text
maxContextLoad
```

迁移规则：

1. 如果存在 `maxContextTokens`，优先使用。
2. 如果不存在但存在旧 `maxContextLoad`，沿用旧值作为 token 阈值并写入新 key。
3. 后续只写 `maxContextTokens`。

用户输入解析规则：

- `250000` => `250000 tokens`
- `250000 tokens` => `250000 tokens`
- `128k` => `131072 tokens`
- `256k` => `262144 tokens`
- `25w` / `25万` => `250000 tokens`
- `1m` => `1048576 tokens`
- `25万字` / `25w chars` / `250000 字符` => 先按字符预算理解，再根据当前模型 profile 换算成 tokens。

也就是说，`k`、`m` 默认按模型上下文窗口常用的 1024 进位解释，`w`、`万` 按十进制万解释；只有用户明确写了 `字`、`字符`、`chars`、`characters` 时，才当作字符预算换算成 tokens。

## 边界场景处理记录

### 1. 开始的时候

没有真实 anchor。

处理：

- 仪表盘走 full estimate。
- 第一次响应如果返回 usage，则建立 anchor。
- 后续开始使用 anchor + delta。

### 2. 倒回的时候

用户切回旧消息，当前链路变短。

处理：

- `Navbar.svelte` 通过 `createMessagesList(history, history.currentId)` 只拿当前分支链路。
- `findValidAnchor()` 沿当前链路向上找最近 anchor。
- 如果 anchor 不在当前链路上，自动不可用。
- 如果找不到可用 anchor，回退 full estimate 或 calibrated estimate。

### 3. 开辟不同分支

同一聊天可能从旧消息生成新分支。

处理：

- 每条 assistant response 都可以有自己的 anchor。
- 切分支时，当前 `history.currentId` 变化，当前 message chain 也变化。
- anchor 查找只在当前链路上进行。
- 其他分支的 anchor 不会误用。

### 4. 开关记忆防爆阀

该开关如果影响实际 payload，旧 anchor 可能不再准确。

处理：

- 将 memory 状态纳入 `settingsHash`。
- hash 变化后，旧 anchor 自动失效。
- 失效后回退 calibrated estimate 或 full estimate。

### 5. 开关记忆抽屉中某一个记忆卡片

启用 / 卸载单张卡片会改变 `manualMemoryText`。

处理：

- MemoryDrawer 每次 `saveCards()` 都会重新 `compileCardsToText(cards)`。
- Navbar 估算时读取最新 `$manualMemoryText`。
- anchor + delta 使用当前 Vault token 与 anchor 时 Vault token 的差值修正。

### 6. 改写、编辑记忆卡片

卡片 title / content 改变会改变 Vault 文本。

处理：

- 同开关卡片逻辑。
- 新 Vault 文本重新估算。
- 使用 `vaultDelta` 修正当前 token load。

### 7. 导入、删除记忆卡片

导入或删除卡片同样会改变 Vault 文本。

处理：

- 重新编译 Vault 文本。
- 重新估算 Vault token。
- anchor 可用时走 `vaultDelta`；anchor 不可用时走全量估算。

### 8. 切换模型

不同模型 tokenizer 不能混用。

处理：

- anchor 中保存 `modelId`。
- `findValidAnchor()` 要求 anchor modelId 与当前 modelId 一致。
- 不一致则跳过 anchor。
- MemoryDrawer 和 Navbar 都使用当前选中模型 profile。

### 9. 修改 system prompt

system prompt 改变会影响实际 prompt。

处理：

- anchor 记录 system prompt 估算值。
- 当前 system prompt 改变时，通过 `systemDelta` 修正。
- 同时 `settingsHash` 用于处理会改变统计口径的设置项。

### 10. strip thinking 开关

开启 / 关闭 reasoning 剥离会改变 assistant 消息计入口径。

处理：

- `stripReasoning` 纳入 `settingsHash`。
- hash 变化后旧 anchor 不复用。

### 11. 文件 / RAG / web search / 工具调用

这些请求会引入前端不可见内容。

处理：

- 本次真实 usage 可以保留在 message 上。
- 不建立 anchor。
- 不更新 calibration。

## 当前实现的准确性边界

已经比旧实现更准确的地方：

- 不再把字符数当 token。
- 不再使用 UTF-16 `.length` 作为统计基础。
- 根据当前模型 profile 区分中文 token 成本。
- 支持真实 usage anchor。
- 支持 anchor + delta，长上下文误差不会简单随总长度线性累积。
- 支持模型 calibration。
- 支持倒回、分支、记忆卡片开关/编辑/导入/删除等关键场景。

仍然不是绝对真实 tokenizer 的地方：

- 第一版没有接入模型真实 tokenizer。
- 对文件、RAG、web search、工具注入内容不强行计入前端仪表盘。
- 后端 filter/outlet 如果隐式改写 prompt，而前端不可见，则只能通过 anchor 失效或后续 usage 校准修正。
- 不同 provider 的 message template overhead 仍是 profile 级近似。

## 后续建议

第一阶段已经完成“字符数仪表盘”到“相对准确 token 仪表盘”的核心升级。

后续如果要继续提高准确性，可以考虑：

1. 为常用本地模型接入轻量 tokenizer，按模型精确计数。
2. 后端暴露 dry-run token count API，由前端节流调用。
3. 对 Ollama 的 `prompt_eval_count` 做更系统的 provider 适配。
4. 在调试模式下显示 `estimate.mode`、`calibrationFactor`、`profileId`，但默认 UI 不暴露这些内部信息。
5. 把 anchor / calibration 从 localStorage 迁移到更正式的 client store 或后端用户配置。

## 总结

本次改造完成后，用户看到的 `20W / 25W` 已经按 token 负载口径理解，而不是旧的字符数口径。

计算路径是：

1. 优先使用真实 usage anchor + delta。
2. 没有 anchor 时使用模型校准倍率修正全量估算。
3. 没有校准数据时使用模型 profile 的全量 token 估算。

这使仪表盘在中文长文本、Vault 记忆、分支对话和长历史场景下比旧字符统计明显更可靠。
