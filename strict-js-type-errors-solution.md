# Strict JS 类型错误治理方案

## 背景

当前项目的 `tsconfig.json` 开启了 `allowJs: true`、`checkJs: true`、`strict: true`。这会让 TypeScript/Svelte 对 `.js`、`.svelte` 文件做严格类型检查。现状是仓库里存在不少历史类型错误，所以 `npm run check` 不能作为干净的通过信号。

这些错误不等同于“应用现在一定会坏”。很多属于类型系统无法理解旧 JS 动态写法，例如隐式 `any`、空对象属性访问、动态 DOM 属性、Svelte 模板规范告警等。但它们会让 CI/本地检查失败，也会掩盖真正的新问题。

## 核心原则

最重要原则：不影响当前稳定可用版本。

治理应按“先隔离、再修复、最后收紧”的顺序推进，避免一次性大改。任何阶段都不应重构聊天、生成、文件导航、记忆、模型选择等核心功能，除非有明确测试和回滚点。

## 建议路线

### 1. 固定当前稳定基线

先保留当前可用版本作为回滚点：

```bash
git status
git tag stable-before-typecheck-cleanup
git push origin stable-before-typecheck-cleanup
```

如果之后类型治理影响使用，可以直接回到这个 tag 或对比差异。

### 2. 建立错误清单，不马上修代码

先把 `npm run check` 输出按文件归类，分成三类：

- 真实潜在 bug：例如可能为 `null` 的 DOM 节点直接访问、数组元素结构不明确。
- 类型债务：例如隐式 `any`、JS 参数没 JSDoc、动态对象属性。
- 工具/环境问题：例如 Windows 下 `esbuild spawn EPERM`。

目标是先知道错误分布，避免盲目全局修改。

### 3. 先处理“检查噪音”，不碰运行行为

优先做不改变运行逻辑的修复：

- 给 JS 文件补 JSDoc 类型。
- 给动态 DOM 属性补局部类型声明。
- 把 `<div />` 改为 `<div></div>` 这类模板等价修复。
- 对明显是旧版动态插件、短期不适合类型化的文件，考虑局部 `// @ts-nocheck`，但要记录原因。

这一步的目标是减少检查噪音，不改变用户功能。

### 4. 再处理高风险文件

按使用频率和风险排序：

1. 聊天消息渲染相关组件。
2. 输入框、富文本、快捷命令组件。
3. 文件导航和代码块渲染组件。
4. 设置页、管理页等低频界面。

每次只处理少量文件，提交粒度要小，例如：

```bash
git commit -m "typecheck: annotate rich text autocomplete"
git commit -m "typecheck: fix message template diagnostics"
```

这样任何问题都容易定位和回滚。

### 5. 每一批都做运行验证

每批修改后至少验证：

- 应用能启动。
- 旧聊天能打开。
- 手机尺寸下能上下滚动历史记录。
- 能发送消息、继续生成、停止生成。
- 代码块、折叠、复制、编辑消息不异常。

类型检查通过不是唯一目标，不能牺牲当前使用体验。

## 不建议的做法

- 不建议一次性把所有 `.js` 改成 `.ts`。
- 不建议大规模格式化整个仓库。
- 不建议为了让 `npm run check` 变绿而顺手重构核心聊天逻辑。
- 不建议关闭全项目严格检查后就不再跟进，因为以后新问题会更难发现。

## 推荐的最终状态

最终目标是：

- `npm run check` 至少不被历史噪音淹没。
- 新改动能被类型检查有效拦住。
- 核心功能行为不变。
- 每一类临时豁免都有注释和后续清理入口。

如果要最稳妥推进，可以先创建单独分支 `codex/typecheck-cleanup`，逐批修复并在本地验证后再合并到 `main`。
